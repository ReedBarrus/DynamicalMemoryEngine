# Dynamical Memory Engine — C1 Status Lifecycle

## Status

This document is a Door Two operational-seam note.

It is not constitutional authority.

It does **not** override:

- `README_MasterConstitution.md`
- `README_ConstitutionAppendix.md`
- `README_WorkflowContract.md`
- `README.DoorTwoCanonCandidatePacket.md`
- `README.DoorTwoCanonActivationCriteria.md`

Its purpose is narrower:

- define the minimal status lifecycle for early C1 objects,
- distinguish review status from canon status,
- define how a C1 claim may change state without silent authority drift,
- preserve governability, reversibility, and scope discipline for the first live canon objects,
- and prevent canon from behaving as sticky truth merely because it was once promoted.

This note is intended for the first live Door Two canon surfaces only.

It should remain:
- small,
- explicit,
- reversible,
- and below any broad canon ontology.

---

## 1. Constitutional posture

Canon remains a distinct promotion-only boundary.

It does not arise automatically from:

- runtime stability,
- recurrence,
- replay success,
- packet existence,
- readiness posture,
- review convenience,
- retention duration,
- or substrate organization. 

Runtime is not canon.
Query is not truth.
Substrate is not ontology.
Consensus is promotion-only. 

Accordingly, C1 lifecycle states must not be treated as:
- runtime moods,
- packet convenience labels,
- or implicit truth gradients.

A lifecycle state exists to govern:
- whether a canon claim is only proposed,
- actively trusted,
- contested,
- narrowed,
- superseded,
- revoked,
- or suspended from use.

---

## 2. Why this note exists

Door Two now has:

- candidate packets,
- activation criteria,
- trial review outcomes,
- one promotable narrow anchor candidate,
- one deferred governability-edge candidate,
- and one rejected anti-pattern. 

That is enough to make one new practical problem real:

**how does a canon claim change status without becoming sticky, vague, or silently over-authoritative?**

Without a lifecycle note, the project risks:

- treating review posture as promotion,
- treating promotion as permanent,
- treating challenge as philosophy rather than operation,
- or losing the difference between narrowing, supersession, deferral, suspension, and revocation.

This note exists to prevent that.

---

## 3. Core lifecycle rule

A C1 object is not governed only by whether it exists.

It is governed by:
- its current status,
- its allowed-use posture,
- its contestability posture,
- and its scope.

**Canon must remain governable after promotion, not only before it.**

This means:

- promotion does not end review forever,
- challenge does not automatically mean revocation,
- narrowing is a lawful status transition,
- suspension is distinct from contest annotation,
- and supersession is distinct from failure.

---

## 4. Scope of this lifecycle

This lifecycle governs:

- early C1 claim objects,
- especially narrow same-family, same-lens canon anchors,
- and other future small-scope canon objects admitted under the current Door Two criteria.

This note does **not** yet define:

- broad canon class hierarchies,
- prediction-facing canon dynamics,
- symbolic canon bindings,
- cross-family canon webs,
- or ontology-bearing canon lattices.

Those remain later concerns.

## 5. Review status vs canon status

A packet and a canon claim are not the same thing.

### 5.1 Packet status
Packet status describes the proposal container.

Examples:
- proposed
- under_review
- deferred
- rejected
- promoted
- superseded
- revoked

These belong to the packet/review surface. :contentReference[oaicite:5]{index=5}

### 5.2 Canon status
Canon status describes the standing of the promoted claim itself.

The present note defines the minimal early C1 canon statuses.

A packet may be deferred while no canon exists.
A canon may be contested while the originating packet remains historically promoted.
A canon may be narrowed without being revoked.

Do not collapse packet history and canon standing into one field.

---

## 6. Minimal early C1 status set

The minimal status set for early C1 should be:

- `proposed`
- `deferred`
- `promoted`
- `contested`
- `suspended`
- `narrowed`
- `superseded`
- `revoked`

This set is intentionally small.

It exists to cover:
- pre-promotion review
- active use
- challenge
- temporary containment
- scope reduction
- replacement
- full withdrawal

---

## 7. Status definitions

## 7.1 `proposed`

Meaning:
A canon candidate exists as a packet or review object but has not yet been promoted.

Allowed use:
- review only
- discussion only
- packet comparison only

Must not imply:
- trusted reuse
- runtime consultation
- standing reference authority

Typical source:
new candidate packet under first review

---

## 7.2 `deferred`

Meaning:
The candidate is reviewable and may be promising, but is not yet promotable under current evidence, scope, or governability conditions.

Allowed use:
- review-only
- design pressure
- future re-review candidate

Must not imply:
- rejection
- promotion
- trusted runtime consultation

Typical reason:
- cross-lens evidence insufficient
- governability not yet sufficient
- claim still needs narrowing
- allowed use still too broad

This is the correct status for useful-but-not-yet-safe candidates such as governability-edge cases.

---

## 7.3 `promoted`

Meaning:
The bounded claim has passed legitimacy, utility, and governability review and is now an active C1 reference surface under declared scope.

Allowed use:
- only the explicitly authorized use surface
- only under the declared scope and lens conditions

Must not imply:
- universality
- ontology
- prediction authority by default
- immunity from challenge
- scope expansion by convenience

Promotion is a trust uplift, not a truth coronation. :contentReference[oaicite:6]{index=6}

---

## 7.4 `contested`

Meaning:
A challenge has been raised against an active C1 claim, but the claim has not yet been removed from use.

Allowed use:
- may continue only under the currently declared contest posture
- must visibly carry contested status wherever consulted

Must not imply:
- automatic revocation
- automatic suspension
- automatic invalidity

A contested canon is still active, but no longer cleanly settled.

This status exists so challenge can be real without forcing immediate binary collapse.

---

## 7.5 `suspended`

Meaning:
A previously promoted C1 claim is temporarily removed from active use pending review, even though it has not yet been fully revoked.

Allowed use:
- historical inspection
- review and contest resolution only

Must not be used for:
- active trusted reference consultation
- runtime handoff
- downstream operational reliance

Typical reason:
- challenge is serious enough that continued use is unsafe
- evidence conflict has crossed a predefined trigger
- blast radius if wrong is no longer acceptable during review

This status is distinct from `contested`.

A contested canon may still be usable.
A suspended canon is not.

---

## 7.6 `narrowed`

Meaning:
The canon claim remains active, but its scope has been reduced.

This may include narrowing:
- source-family scope
- cohort scope
- lens scope
- temporal scope
- allowed downstream use

Allowed use:
- only under the revised narrower scope

Must not imply:
- replacement by a new claim
- full revocation
- mere annotation

This is a critical early Door Two status because many canon claims will likely survive only by shrinking.

Narrowing is a governability action, not a cosmetic note.

---

## 7.7 `superseded`

Meaning:
A previously active canon claim has been replaced by a stronger or more recent canon claim.

Allowed use:
- historical reference only
- lineage inspection
- comparison against newer canon if relevant

Must not imply:
- the prior claim was illegitimate when promoted
- the prior claim is revoked for being false

Supersession means:
a newer canon claim now governs the same practical role more appropriately.

This is replacement, not failure.

---

## 7.8 `revoked`

Meaning:
A previously promoted canon claim is no longer trusted for its authorized use.

Allowed use:
- historical inspection only
- failure analysis
- review lineage

Must not be used for:
- active trusted reference
- runtime consultation
- downstream operational support

Typical reason:
- later evidence breaks the promoted claim materially
- challenge succeeds strongly enough that trust uplift can no longer stand
- governability or legitimacy assumptions fail in a non-recoverable way

Revocation is stronger than narrowing and stronger than suspension.

---

## 8. Allowed transitions

The earliest lawful transition map is:

```text
proposed
  → deferred
  → promoted
  → rejected

promoted
  → contested
  → suspended
  → narrowed
  → superseded
  → revoked

contested
  → promoted
  → suspended
  → narrowed
  → superseded
  → revoked

suspended
  → promoted
  → narrowed
  → superseded
  → revoked
``` 
Not every transition must be used often.
But these are the minimal lawful paths needed to avoid sticky canon.

# 9. Transition meanings
9.1 proposed → deferred

Use when:

the packet is reviewable
but not yet promotable
9.2 proposed → promoted

Use when:

all three gates pass
and the claim is authorized for bounded use
9.3 promoted → contested

Use when:

a valid challenge is raised
but active use is not yet deemed unsafe enough to stop immediately
9.4 promoted → suspended

Use when:

a challenge or evidence conflict is strong enough that continued active use is unsafe pending review
9.5 promoted → narrowed

Use when:

the claim survives only by shrinking scope
and this is cleaner than revocation or supersession
9.6 promoted → superseded

Use when:

a later, stronger canon claim now properly takes over the same role
9.7 promoted → revoked

Use when:

the claim no longer deserves trusted use
9.8 contested → promoted

Use when:

review resolves the challenge and active status is restored without narrowing
9.9 contested → narrowed

Use when:

the challenge succeeds partially
and reduced scope resolves the risk
9.10 contested → suspended

Use when:

challenge pressure becomes too strong for continued safe active use
9.11 suspended → promoted

Use when:

review clears the issue and active use resumes
9.12 suspended → narrowed

Use when:

active use may resume only under reduced scope
9.13 suspended → revoked

Use when:

review determines the claim can no longer be trusted

# 10. Status behavior rules
Rule 1 — status must be explicit

No C1 object may silently change standing through implication.

Rule 2 — status does not erase lineage

A superseded or revoked canon still retains historical review lineage.

Rule 3 — contested is not revoked

Challenge must not automatically collapse into failure.

Rule 4 — suspended is not merely annotated

Suspended canon is removed from active use until review resolves it.

Rule 5 — narrowed is a real state

Scope reduction must be represented explicitly, not hidden in note text only.

Rule 6 — promoted does not mean universal

All active use remains bounded by declared scope and allowed use.

# 11. Minimal status metadata

Every live C1 object should preserve at minimum:

canonical_id
canonical_status
canonical_version
status_reason
effective_scope
allowed_use
challenge_posture
supersession_or_revocation_ref if applicable

This is the minimum needed to keep status machine-legible and governable.

# 12. Relationship to challenge workflow

This note defines status states.

It does not define the full challenge procedure.

That belongs in the separate challenge / contest / supersession workflow note.

However, this note does establish the minimum distinction:

contested = challenged, still active under visible contest posture
suspended = challenged strongly enough that active use stops pending review
revoked = no longer trusted for active use

That distinction must be preserved downstream.

# 13. Relationship to runtime handoff

This note does not yet define how C1 becomes runtime-usable.

That belongs in the activation / runtime handoff boundary note.

However, this note does establish a critical operational rule:

Only promoted, contested, or narrowed canon objects may even be candidates for active consultation, and only under declared allowed-use rules.

deferred, suspended, superseded, and revoked objects must not be treated as active trusted runtime references.

# 14. Earliest recommended status use

For the first live C1 cycle, the expected usage is:

Packet 1 style narrow anchor → promoted
Packet 3 style governability-edge candidate → deferred
anti-pattern overclaim → rejected at packet level, no live C1 object created

Early use should stay this simple.

Do not expand the lifecycle just because more statuses are imaginable.

# 15. Working summary

The first lawful C1 lifecycle should stay small and operational.

Its purpose is to make canon:

explicitly governable
challengeable
narrowable
suspendable
supersedable
revocable

without collapsing all canon change into either “active forever” or “fully dead.”

The key early insight is simple:

some canon survives by narrowing, some by challenge resolution, some by supersession, and some should stop being used entirely.
