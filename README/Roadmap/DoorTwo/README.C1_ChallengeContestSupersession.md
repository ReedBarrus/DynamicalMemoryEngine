## 1. Constitutional posture

Canon remains a distinct promotion-only boundary.

It does not arise automatically from:
- runtime stability,
- recurrence,
- replay success,
- packet existence,
- retention duration,
- readiness posture,
- or review convenience.

Runtime is not canon.
Query is not truth.
Substrate is not ontology.
Consensus is promotion-only.

Accordingly, challenge handling must not:
- collapse canon back into runtime,
- treat contest as ontology failure by default,
- silently expand canon scope during review,
- or let preservation/review surfaces masquerade as authority.

Challenge exists to preserve governability after promotion.

---

## 2. Why this note exists

The current Door Two review structure now has:

- one promotable narrow anchor,
- one deferred governability-edge candidate,
- one rejected anti-pattern,
- and a status lifecycle that distinguishes promoted, contested, suspended, narrowed, superseded, and revoked.

That is enough to make the next practical question real:

**What happens when an active C1 claim is challenged?**

Without a challenge workflow, the project risks:
- treating contestability as symbolic rather than operational,
- collapsing every challenge into either “ignore it” or “revoke it,”
- failing to distinguish annotation from temporary suspension,
- and leaving supersession, narrowing, and revocation too vague for actual use.

This note exists to prevent that.

---

## 3. Core rule

A challenge does not decide canon by itself.

A challenge does three smaller things:

1. raises a bounded question about an active canon claim,
2. forces review of that question under explicit scope,
3. may change status only through an explicit review outcome.

This means:

- challenge is not automatic revocation,
- challenge is not automatic suspension,
- challenge is not mere commentary,
- and challenge must remain tied to a declared claim, scope, and trigger.

---

## 4. Scope

This note governs early C1 challenge handling for:

- narrow same-family, same-lens canon anchors,
- other future small-scope C1 claims admitted under the current Door Two criteria.

It does **not** yet define:

- broad canon court structure,
- prediction-facing canon arbitration,
- symbolic challenge layers,
- agentic dispute handling,
- or many-class canon governance.

Those remain later concerns.

---

## 5. Minimal challenge object

Every challenge to an active C1 should minimally preserve:

- `canonical_id`
- `challenge_id`
- `challenge_type`
- `challenge_trigger`
- `challenged_claim_scope`
- `reason_for_challenge`
- `current_status_before_challenge`
- `recommended_interim_posture`
- `review_required`

A challenge should be attached to a canon claim.
It should not silently rewrite the canon claim inline.

---

## 6. Challenge types

The minimum early challenge types are:

### 6.1 Evidence challenge
Used when later evidence appears to weaken or contradict the promoted claim under its declared scope.

Examples:
- same-family evidence no longer preserves the anchor behavior
- a return-like relationship no longer reproduces under the same lens
- claimed recurrence weakens materially

### 6.2 Lens challenge
Used when the promoted claim appears too dependent on one lens or breaks under another declared lens.

Examples:
- cross-lens comparison materially changes the claim
- the current lens appears to be carrying the claim too strongly
- lens-relative scope was underappreciated

### 6.3 Scope challenge
Used when the claim is being used more broadly than its declared scope safely permits.

Examples:
- same-family anchor used cross-family
- review anchor treated as event truth
- narrow baseline reference used as general ontology

### 6.4 Governability challenge
Used when the canon claim remains evidentially plausible but begins behaving unsafely as a standing reference surface.

Examples:
- authority gravity becomes too strong
- later users overread the claim repeatedly
- contestability is present in theory but failing in practice

### 6.5 Attachment challenge
Used when canon-ineligible supporting surfaces begin to be treated as if they were canon by attachment.

Examples:
- digest phrasing treated as canonical claim content
- HUD packaging treated as authority
- support descriptors treated as promoted truth objects

These types are intentionally small and early-stage.

---

## 7. Challenge triggers

A challenge must name at least one concrete trigger.

Allowed trigger classes include:

- later same-scope evidence conflict
- failure to reproduce the claim under the same declared lens
- cross-lens instability
- repeated scope misuse
- review-discovered attachment leakage
- materially increased blast radius if the claim stays active

A challenge without a concrete trigger should remain commentary, not workflow.

---

## 8. Interim postures

Once a valid challenge is raised, review must choose an interim posture.

The minimum early postures are:

### 8.1 Annotate as contested
Use when:
- the challenge is real,
- but continued use is still acceptable during review.

Effect:
- canon remains active,
- but every consultation must visibly preserve contested status.

Maps to lifecycle:
- `promoted → contested`

### 8.2 Suspend active use
Use when:
- the challenge is serious enough that continued active use is unsafe pending review.

Effect:
- canon remains historically present,
- but may not be used as an active trusted reference until review resolves it.

Maps to lifecycle:
- `promoted → suspended`

### 8.3 Keep active pending evidence collection
Use when:
- challenge is legitimate,
- but current evidence is too thin to justify contest or suspension yet.

Effect:
- no status change yet,
- but review is now open and evidence collection is required.

This posture should be used sparingly.

---

## 9. Review outcomes after challenge

A completed challenge review may result in:

### 9.1 Restore promoted status
Use when:
- the challenge does not materially break the claim,
- and the claim remains governable as-is.

Maps to lifecycle:
- `contested → promoted`
- or `suspended → promoted`

### 9.2 Narrow the claim
Use when:
- the claim survives only by reducing scope,
- reducing allowed use,
- or tightening lens/family/temporal bounds.

Maps to lifecycle:
- `promoted → narrowed`
- `contested → narrowed`
- `suspended → narrowed`

Narrowing is not a cosmetic patch.
It is a genuine governance outcome.

### 9.3 Supersede the claim
Use when:
- a stronger canon claim now fills the same role more lawfully.

Maps to lifecycle:
- `promoted → superseded`
- `contested → superseded`
- `suspended → superseded`

Supersession is replacement, not failure.

### 9.4 Revoke the claim
Use when:
- the claim no longer deserves trusted use,
- and neither narrowing nor supersession is the honest remedy.

Maps to lifecycle:
- `promoted → revoked`
- `contested → revoked`
- `suspended → revoked`

Revocation is stronger than challenge, stronger than suspension, and stronger than narrowing.

### 9.5 Keep deferred below canon
Use when:
- review reveals the object should no longer remain active as canon,
- but a revised candidate may later re-enter review as a packet.

This outcome should typically be represented by:
- revoking or superseding the active canon claim,
- while preserving a packet or review object below canon.

---

## 10. Distinguishing contest, suspension, narrowing, supersession, revocation

### Contest
A challenge is active.
The claim is still active, but visibly unsettled.

### Suspension
A challenge is active.
The claim is temporarily removed from active use pending review.

### Narrowing
The claim survives by shrinking scope.

### Supersession
The claim is replaced by a stronger canon claim serving the same role.

### Revocation
The claim is no longer trusted for active use.

These are materially different and must not be collapsed.

## 11. Minimal workflow

### Step 1 — Raise challenge
Record:
- challenged canon object
- challenge type
- challenge trigger
- reason
- recommended interim posture

### Step 2 — Determine interim posture
Choose one:
- annotate as contested
- suspend active use
- keep active pending evidence collection

### Step 3 — Review under declared scope
Review must inspect:
- whether the original claim still holds,
- whether the scope is still lawful,
- whether the allowed-use surface remains safe,
- whether the challenge is evidence, lens, scope, governability, or attachment dominant,
- and whether blast radius remains acceptable.

### Step 4 — Resolve
Choose one:
- restore promoted status
- narrow
- supersede
- revoke

### Step 5 — Preserve lineage
A challenge must not erase:
- original promotion record
- prior scope
- reason for challenge
- final outcome

Canon history must remain inspectable.

---

## 12. Challenge behavior rules

### Rule 1 — Challenge must be scoped
A challenge must target a bounded claim under declared conditions.

### Rule 2 — Challenge must be triggered
A challenge must name a concrete trigger, not only discomfort.

### Rule 3 — Contest is visible
If a canon object is contested but still active, its contested posture must remain visible wherever consulted.

### Rule 4 — Suspension halts active use
Suspended canon must not be used as an active trusted reference.

### Rule 5 — Narrowing must update scope explicitly
A narrowed canon must preserve updated:
- claim scope
- allowed use
- non-claims
- and status reason

### Rule 6 — Supersession preserves history
A superseded canon is not deleted.
It is historically preserved and operationally replaced.

### Rule 7 — Revocation is explicit
Revoked canon must remain visible as revoked, not silently disappear.

### Rule 8 — Attachment leakage is challengeable
If a canon-ineligible supporting surface starts behaving like canon, that is a valid challenge.

## 13. Relationship to activation criteria

This note does not redefine legitimacy, utility, or governability.

It operationalizes what happens **after** promotion when those surfaces are pressured again.

The activation criteria remain the gate for admission.
This note defines the smallest lawful loop for:
- challenge,
- contest,
- suspension,
- narrowing,
- supersession,
- and revocation.

In particular, this note carries forward the governability principle:
a canon claim must remain bounded, contestable, and non-distorting as a standing reference surface.

## 15. Earliest expected use

For the first live C1 cycle, the expected use is simple:

- Packet 1 style narrow baseline anchor may be challenged if same-family / same-lens anchor behavior later breaks or if scope misuse appears
- Packet 3 style deferred candidate remains below canon and therefore does not yet enter active challenge workflow
- rejected anti-pattern packets remain outside canon entirely

This note should stay this boring until those first cases actually occur.

## 17. Working summary

A challenge does not automatically kill canon.

But it must be able to:
- visibly contest it,
- suspend it if needed,
- narrow it when that is the honest fix,
- supersede it when a stronger claim takes over,
- or revoke it when trusted use can no longer stand.

That is the smallest lawful post-promotion governance loop for early C1.
