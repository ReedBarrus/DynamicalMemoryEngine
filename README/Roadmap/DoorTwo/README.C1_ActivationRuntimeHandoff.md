# Dynamical Memory Engine — C1 Activation / Runtime Handoff Boundary

## Status

This document is a Door Two operational-seam note.

It is not constitutional authority.

It does **not** override:

- `README_MasterConstitution.md`

Its purpose is narrower:

- define when a promoted C1 claim becomes runtime-usable at all,
- define the smallest lawful handoff from review/promotion into bounded consultation,
- preserve strict separation between runtime memory, substrate memory, and canonical memory,
- define which C1 statuses may be consulted,
- define which surfaces may consult C1,
- define which surfaces must not consult C1,
- and make early C1 allowed-use boundaries explicit enough to resist scope creep.

This note is intended for the first live Door Two canon surfaces only.

It should remain:
- small,
- explicit,
- machine-legible where practical,
- and below prediction, policy, symbolic, or broad ecology concerns.

## 1. Constitutional posture

Canon remains a distinct promotion-only boundary.

It does not arise automatically from:
- runtime stability,
- recurrence,
- replay success,
- packet existence,
- retention duration,
- readiness posture,
- review convenience,
- or substrate organization. 

Runtime is not canon.
Query is not truth.
Substrate is not ontology.
Consensus is promotion-only. 

Accordingly, runtime handoff must not:

- collapse canon back into runtime memory,
- let review packaging masquerade as active C1,
- treat promotion as universal consultation authority,
- silently import prediction or policy semantics,
- or allow canon-ineligible supporting surfaces to become active by attachment.

Handoff is not ontology transfer.
Handoff is bounded consultation permission.

---

## 2. Why this note exists

Door Two now has enough structure to promote a narrow first canon candidate:
- candidate packets,
- activation criteria,
- a live status lifecycle,
- a challenge / contest / supersession workflow,
- one credible narrow promotion lane,
- one deferred governability-edge lane,
- and one rejected anti-pattern. 

That creates the next practical question:

**When does a promoted C1 claim become usable by the system, and under what exact restrictions?**

Without a handoff note, the project risks:
- treating promotion memos as if they were already active runtime authority,
- letting any active canon object be consulted everywhere by default,
- blurring packet review, canonical memory, and runtime memory,
- and using C1 more broadly than its allowed scope actually permits.

This note exists to stop that drift.

---

## 3. Core handoff rule

A promoted C1 claim is not automatically globally usable.

A promoted C1 claim becomes runtime-usable only when all three of the following are true:

1. the claim is in a consultable canon status,
2. the allowed-use surface explicitly permits the proposed consultation,
3. the consultation occurs through a lawful handoff seam that preserves scope visibility.

If any of the three are false, the C1 object must not be used as an active trusted reference.

---

## 4. Handoff principle

The purpose of early C1 handoff is:

- not to reshape runtime,
- not to bypass replay evidence,
- and not to authorize broad prediction.

The purpose is narrower:

**allow a small promoted canon claim to act as a bounded trusted reference surface for explicitly whitelisted comparison tasks.**

Early C1 is therefore best understood as:
- a bounded trust anchor,
- not a general intelligence substrate,
- not a universal memory layer,
- not a semantic object,
- and not an automatic decision input.

---

## 5. Separation rule

The following must remain distinct:

### Runtime memory
Replayable compressed structural states from the pipeline. Examples: `H1`, `M1`. 

### Substrate memory
Organized ecology over runtime memory: segments, trajectories, proto-basins, neighborhood structures. 

### Canonical memory
Promoted trusted memory admitted under explicit legitimacy criteria. `C1`. 

C1 handoff must never:
- rewrite runtime memory,
- overwrite substrate history,
- or cause substrate organization to be treated as canon by proximity.

C1 may only be consulted as an external trusted reference surface under declared scope.

---

## 6. Consultable status rule

Only some C1 statuses are even eligible for active consultation.

### 6.1 Consultable statuses

The only statuses that may be candidates for active consultation are:

- `promoted`
- `contested`
- `narrowed`

Even then, consultation is allowed only if:
- the allowed-use surface permits it,
- and the consultation posture preserves the current status visibly and materially. 

### 6.2 Non-consultable statuses

The following statuses must not be used as active trusted runtime references:

- `proposed`
- `deferred`
- `suspended`
- `superseded`
- `revoked`

These may still be:
- inspected historically,
- reviewed,
- compared,
- or cited as lineage,

but they must not function as active trust anchors. :contentReference[oaicite:9]{index=9}

---

## 7. Consultability meanings by status

### 7.1 `promoted`
Fully consultable within declared allowed use.

### 7.2 `contested`
Consultable only if:
- contested posture is made visible at consultation time,
- the allowed use remains narrow,
- and the challenge workflow has not moved the object to suspension.

Contested canon must not look identical to clean promoted canon during consultation. :contentReference[oaicite:10]{index=10}

### 7.3 `narrowed`
Consultable only under the revised narrower scope.

Any prior broader scope must be treated as inactive.

---

## 8. Early C1 allowed-use whitelist

For the first live C1 cycle, allowed use should be machine-legible and whitelist-like.

### 8.1 Example allowed-use fields
Every live C1 object should preserve explicit fields such as:

- `allowed_use`
- `forbidden_use`
- `consultation_scope`
- `consultation_status_visibility`
- `runtime_handoff_enabled`

These fields exist to prevent later ambiguity.

### 8.2 First live C1 whitelist posture
For the first Packet 1 style baseline anchor, the preferred whitelist is:

**May be consulted for:**
- same-family baseline comparison
- same-family perturbation-vs-baseline comparison
- same-family narrow return-vs-perturb reference
- bounded review-anchor consultation under the same declared lens

**May not be consulted for:**
- cross-family comparison by default
- undeclared-lens comparison
- prediction
- policy or control input
- ontology inference
- event truth
- symbolic binding authority
- semantic label generation as if canon were language

This whitelist should remain explicit in the live C1 object itself, not merely in prose notes.

---

## 9. Surfaces allowed to consult C1

The first live C1 consultation surfaces should be minimal.

### 9.1 Allowed early consultation surfaces

#### A. Door Two review / memo surfaces
May consult C1 directly because they are part of the canon review and governance loop.

#### B. Bounded comparison / review workflows
May consult C1 when the purpose is:
- same-family anchor comparison,
- bounded review reuse,
- or narrow continuity/reference checking under declared scope.

#### C. Future bounded handoff helpers
A small dedicated canon consultation helper or read-only canon store may consult active C1 when it preserves:
- status visibility,
- scope visibility,
- and allowed-use checks.

### 9.2 Important note
Allowed consultation does not mean broad runtime mutation.
It means bounded read access for declared trust-anchored comparison.

---

## 10. Surfaces forbidden to consult C1 directly

The following surfaces must not directly treat C1 as active authority in the first live cycle:

### A. Door One ingest seams
Pre-ingest adapters, ingest contracts, and executive acceptance gates remain below canon and must not consult C1. 

### B. Door One structural/runtime operators
Core runtime operators must not silently consult canon in the first live C1 cycle. This includes transforms, compression, anomaly, merge, reconstruct, and substrate components unless explicitly authorized in a later Door Two step. 

### C. HUD / inspection surfaces as if they were authority surfaces
HUDs may display C1 posture if later authorized, but they must not decide or infer canon semantics from display convenience. Door One inspection remains provenance-first and below canon. 

### D. Prediction / policy / agency surfaces
Prediction, policy, and control use are not part of the first live C1 handoff. Those remain later-door concerns. 

### E. Canon-ineligible supporting surfaces
Digest language, UI packaging, convenience summaries, or support descriptors must not become active canon by attachment.

## 11. Minimal handoff seam

The first live handoff should be boring.

A lawful early handoff should look like:

```text
promoted C1 claim
→ explicit allowed-use check
→ explicit status check
→ explicit scope/lens check
→ bounded consultation result
```

This means the handoff seam should enforce at minimum:

current canon status
allowed-use whitelist
forbidden-use list
source-family scope
lens scope
consultation visibility posture

If any one of these is missing, handoff should fail closed.

# 12. Fail-closed rule

A C1 object must not be consulted when:

status is non-consultable,
allowed-use is absent,
forbidden-use is absent,
scope is underdeclared,
current task exceeds the whitelisted use,
status visibility cannot be preserved,
or challenge/suspension posture makes active use unsafe.

Early C1 should fail closed rather than drift open.

# 13. Status visibility at handoff time

A consultation result must preserve the active canon status visibly enough that later users do not mistake:

contested for clean promoted,
narrowed for original broad scope,
or suspended for still-active canon.

At minimum, any consultation surface should preserve:

canonical_id
canonical_status
effective_scope
allowed_use
forbidden_use
status_reason

This is especially important for contested objects. A contested canon may be consultable, but only if the contested posture remains unmissable.

# 14. Minimal live C1 fields required for handoff

Before a C1 object is runtime-usable at all, it should minimally preserve:

canonical_id
canonical_version
canonical_status
promoted_claim
effective_scope
allowed_use
forbidden_use
status_reason
challenge_posture
runtime_handoff_enabled

These are the minimum fields needed to make consultation governable and machine-legible.

# 15. First live C1 posture

For the first live cycle, Packet 1 style promotion should be treated as:

one same-family
one same-lens
baseline-anchor claim
with narrow allowed-use whitelist
and explicit forbidden-use list

That means the first live C1 should not be used as:

a broad reference lattice
a semantic anchor
a prediction object
a cross-family regime classifier
or a symbolic attachment hub

One tiny anchor and nothing else.

# 16. Relationship to challenge workflow

This note depends directly on the challenge / contest / supersession workflow.

A challenged canon may only remain consultable if:

challenge posture remains visible,
no suspension has occurred,
and the allowed-use whitelist still permits the consultation.

If challenge pressure is strong enough to suspend active use, handoff must stop immediately.

# 17. Relationship to preservation and review materials

Promotion memos, defer memos, review summaries, pinned packets, digests, and archive bundles may all support canon review and lineage.

They do not become consultable canon merely by existing. Preservation remains preservation-only, and packet/review material remains below active canon unless an explicit handoff activates a bounded C1 claim.

# 18. Relationship to future runtime design

This note does not yet define:

a final canon store,
a final ConsensusOp runtime implementation,
a general C1 query engine,
prediction-handle consultation,
or broad system-wide canon routing.

Those may come later.

This note only defines the smallest lawful early handoff for one live C1 object.

# 19. Practical earliest implementation rule

Before any live implementation of C1 consultation, the system should first support:

explicit C1 object storage or registry
explicit status field
explicit allowed-use whitelist
explicit forbidden-use list
explicit scope and lens checks
explicit fail-closed behavior

Do not wire live C1 consultation into broader runtime paths until all six are present.

# 20. Working summary

A promoted C1 claim is not automatically a globally usable runtime authority.

It becomes runtime-usable only when:

its status is consultable,
its allowed use explicitly permits the consultation,
its scope and lens are checked,
its current status remains visible,
and the consultation occurs through a bounded fail-closed handoff seam.

That is the smallest lawful bridge from Door Two review architecture into first live C1 behavior.
