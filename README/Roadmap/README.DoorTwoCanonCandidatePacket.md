## 1. Constitutional posture

Canon remains a distinct promotion-only boundary.

It does not arise automatically from:

- recurrence,
- stability,
- replay success,
- query usefulness,
- substrate organization,
- retention duration,
- packet existence,
- or workbench packaging.

Runtime is not canon.
Query is not truth.
Substrate is not ontology.
Consensus is promotion-only.

Accordingly, a canon candidate packet is:

- review material,
- not promotion,
- not truth,
- not ontological closure,
- not self-authorizing.

Its purpose is to make proposed trust uplift auditable and bounded.

---

## 2. Why this note exists

Door One can now produce:

- replayable structural memory,
- cross-run evidence,
- bounded recognition,
- real-source comparison,
- transition/support descriptors,
- distortion audits,
- and packetized preservation.

That is enough to make canon pressure real.

But it is **not** enough to promote anything safely unless proposed canon claims are first packaged in a disciplined way.

This note exists to force that discipline.

The candidate packet should reveal:

- whether the proposed claim is actually bounded,
- whether Door One evidence is sufficient,
- whether ambiguity is still too strong,
- whether the proposed use is too broad,
- and whether the candidate would behave safely as a future reference surface.

---

## 3. Core packet rule

**Canon should promote bounded claims, not vibes.**

A canon candidate packet must therefore propose:

- a bounded claim,
- under declared conditions,
- with explicit evidence basis,
- explicit non-claims,
- explicit allowed downstream use,
- and explicit review risks.

This means early canon is expected to be:

- narrow,
- scoped,
- auditable,
- reversible,
- and governable.

---

## 4. The three-gate model

Every candidate packet exists to support review across three gates.

### Gate 1 — Legitimacy

Question:

**Can this claim be trusted enough to uplift at all?**

Examples of concerns:
- support basis sufficiency
- ambiguity acceptability
- replay legitimacy
- cross-run evidence
- lens declaration
- family scope honesty

### Gate 2 — Utility

Question:

**If legitimate, is it worth promoting?**

Examples of concerns:
- repeated re-establishment cost
- lawful reuse compression
- packet burden reduction
- stable comparison-anchor value
- downstream usefulness under declared scope

### Gate 3 — Governability

Question:

**If promoted, can it remain safely bounded, contestable, and non-distorting as a standing reference surface?**

Examples of concerns:
- scope inflation risk
- lens-scope forgetfulness
- authority gravity
- blast radius if wrong
- contestability
- revocation / supersession path
- canon-ineligible attachment surfaces

A candidate packet is not complete if it only argues legitimacy and utility while ignoring governability.

---

## 5. Candidate packet object model

A candidate packet should be understood as:

- a **proposal packet**
- carrying a **bounded canon claim**
- attached to **supporting runtime/substrate evidence**
- prepared for **explicit consensus review**
- without self-promoting into canon

This means the promoted primitive is expected to be:

- first: a **bounded claim**
- second: optionally attached to an object, cohort, or reference packet

The packet is a support container.
The claim is the proposed promoted unit.

## 6. Packet fields

Every canon candidate packet should include the following sections.

## 6.1 Packet identity

### `candidate_packet_id`
Stable identifier for the packet.

### `candidate_label`
Short human-readable name.

### `candidate_version`
Version of the packet.

### `candidate_status`
One of:
- proposed
- under_review
- deferred
- rejected
- promoted
- superseded
- revoked

Packet status is not itself canon status.

---

## 6.2 Candidate claim

### `proposed_claim`
A concise sentence describing the exact claim proposed for trust uplift.

Example style:

> Under declared lens L and cohort scope S, reference pattern R is trusted as a stable baseline comparison anchor for use U.

### `claim_type`
Provisional examples:
- baseline_reference_claim
- recurring_transition_claim
- exclusion_claim
- cohort_anchor_claim
- support_attachment_claim
- other_bounded_claim

These are provisional candidate classes, not a fixed canon ontology.

### `claim_scope`
Explicit scope for the proposed claim.

Include:
- source family scope
- cohort scope
- run scope
- lens scope
- temporal scope
- reuse scope

### `explicit_non_claims`
List what the candidate is **not** claiming.

Examples:
- not semantic identity
- not universal across all lenses
- not ontology
- not prediction
- not control policy
- not generalized attractor law

---

## 6.3 Source and lineage surface

### `source_family`
What source family produced the evidence.

### `source_ids`
Source identifiers involved.

### `stream_ids`
Relevant stream identifiers.

### `channel_modality_scope`
Relevant channels/modalities.

### `run_lineage`
Which runs, cohorts, or sessions are included.

### `receipt_refs`
Durable receipt references supporting replay-honest lineage.

### `pinned_packet_refs`
If the candidate depends on preserved packet support.

### `preservation_posture`
What retained tiers support this packet:
- receipts
- digest
- pin
- archive

The packet must remain explicit that preservation does not itself uplift authority.

---

## 6.4 Declared lens

### `declared_lens`
The exact observational lens under which the claim is proposed.

At minimum include:
- nominal sample rate
- effective sample rate / decimation
- transform family
- window family
- window/hop
- band partition
- segmentation rule
- comparison basis
- relevant scale posture
- reconstruction/replay lens if used

### `lens_variation_tested`
Which alternate declared lenses were tested, if any.

### `lens_scope_note`
Whether the proposed claim is:
- single-lens only
- multi-lens repeated
- cross-lens stable
- unresolved across lens variation

Early canon may be lens-scoped.
That scope must remain visible.

---

## 6.5 Evidence basis

### `support_basis`
Compact list of exact evidence surfaces supporting the claim.

Examples:
- recurrence_match
- boundary_peak
- profile_shift
- concentration_support
- basin_split_observed
- weak_support_only
- return_similarity
- cross_run_stability
- replay_legitimacy_confirmed

### `runtime_artifact_refs`
Relevant runtime artifacts:
- A1 / A2 / W1 / S1 / H1 / M1 / A3 / Q / BN
where applicable

### `read_side_probe_refs`
Relevant probes or report surfaces used.

### `cross_run_evidence`
Summary of repeated-run support.

### `real_source_support`
Whether the claim is supported by real-source evidence, synthetic-only evidence, or mixed evidence.

### `preserved_distinctions`
What distinctions the candidate preserves well.

### `collapsed_distinctions`
What distinctions remain flattened or unresolved.

---

## 6.6 Ambiguity and distortion posture

### `ambiguity_posture`
One of:
- low_ambiguity
- bounded_ambiguity
- unresolved_ambiguity
- authority_breaking_ambiguity

### `ambiguity_notes`
Explain what ambiguity remains and whether it affects the claim.

### `distortion_audit_status`
Include:
- last audited layer(s)
- severity
- downstream impact
- recommended action

### `authority_breaking_risks`
List any ambiguity or flattening that would make promotion unsafe.

A candidate with unresolved ambiguity may still be reviewable.
A candidate with authority-breaking ambiguity should not promote.

---

## 6.7 Legitimacy section

### `legitimacy_summary`
Why the claim is trustworthy enough to review for uplift.

### `legitimacy_requirements_met`
Checklist-style section, for example:
- bounded claim
- declared lens
- replay-honest lineage
- support basis sufficient
- ambiguity within acceptable range
- cross-run support adequate
- non-claims explicit

### `failure_risk_if_wrong`
What goes wrong if the claim is trusted but false.

Examples:
- low local comparison error
- baseline misreference
- transition misclassification
- broad downstream interpretive distortion

This is not just confidence.
It is acceptable-error accounting.

---

## 6.8 Utility section

### `allowed_downstream_use`
What later use this canon would authorize.

Examples:
- trusted baseline comparison anchor
- trusted recurring regime reference
- trusted exclusion surface
- bounded anomaly baseline
- bounded reuse in review workflow

### `reuse_compression_benefit`
How much repeated lawful re-establishment this promotion is expected to compress.

Possible posture:
- low
- moderate
- high

Calibration note:

- **moderate** is generally sufficient when the compression benefit recurs across multiple review sessions and the same bounded claim would otherwise require reopening the full packet each time,
- **low** is generally insufficient on its own unless paired with unusually strong legitimacy and very low governability risk.

### `packet_burden_if_not_promoted`
What must be repeatedly reinspected or recomputed if the candidate remains below canon.

### `utility_summary`
Why promotion would materially help rather than merely decorate.

---

## 6.9 Governability section

### `governability_summary`
Why the candidate can remain bounded and safe as a standing reference surface.

### `scope_inflation_risk`
How likely later users are to overread the claim beyond its declared scope.

### `authority_gravity_risk`
How likely the object is to become a shortcut or default truth object.

### `contestability_path`
How future evidence may challenge the claim.

### `revocation_or_supersession_path`
How the claim could be:
- narrowed
- superseded
- revoked
- downgraded to review-only

### `blast_radius_if_misused`
What later surfaces would be distorted if the canon were overtrusted.

### `canon_ineligible_attachments`
Any attached surfaces that must remain non-canonical even if the claim promotes.

Examples:
- support descriptors
- convenience digest language
- unstable summaries
- UI packaging

---

## 6.10 Review decision surface

### `review_recommendation`
One of:
- promote_candidate
- defer_candidate
- reject_candidate
- request_narrower_claim
- request_more_evidence
- request_lower_scope
- request_governability_clarification

### `review_blockers`
Explicit blockers preventing promotion.

### `review_notes`
Bounded notes from review.

### `consensus_scope`
What consensus is being asked to inspect.

Consensus should inspect:
- bounded claim sufficiency
- scope discipline
- ambiguity acceptability
- governability
- allowed use

Consensus is not a generic scoring pass.

# 8. Earliest recommended candidate style

The smallest lawful first canon candidate should probably be:

narrow
claim-level
lens-scoped
strongly replay-supported
low-blast-radius if wrong
clearly useful as a baseline comparison anchor

This means early canon should probably not begin with:

rich semantic regime claims
basin-as-ontology claims
transition mythology
broad universal family claims
support descriptors as standalone canon

The first canon should be small, useful, and safely reversible.

# 9. Canon-ineligible by default

The following should be treated as canon-ineligible by default unless later rules explicitly narrow an exception:

raw runtime summaries
unstable digest language
HUD/display packaging
convenience abstractions
unresolved probe-local heuristics
family-specific support descriptors promoted as standalone truth
substrate organization taken directly as ontology

These may support a candidate packet.
They should not promote directly.

# 10. Trial-packet rule

Before activating Door Two canon, create at least three trial packets:

one likely-pass candidate
one likely-fail candidate
one ambiguous edge case

The ambiguous case is required.

Why:
pass/fail alone will not reveal where ambiguity, scope, and governability rules are still underdefined.

These trial packets should be used to expose:

Door One evidence gaps
packet insufficiencies
ambiguity boundary confusion
overbroad use claims
missing contestability/revocation rules

# 11. Working summary

A canon candidate packet is not a promoted memory object.

It is a bounded proposal packet carrying:

a scoped claim,
explicit evidence basis,
explicit non-claims,
explicit allowed use,
and explicit review risks.

Its purpose is to make canon review disciplined enough that Door Two trust uplift can be debated honestly, narrowly, and reversibly.
