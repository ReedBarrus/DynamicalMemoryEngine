# README.AppSurfaceTransitionPlan.v0.md

## Dynamical Memory Engine — App Surface Transition Plan v0

### Status

This document records the next bounded transition for the DME app surface.

It is a planning and surface-governance note.

It does **not** override:

* `README_MasterConstitution.md`
* `README.WorkflowContract.md`
* `README.StructuralIdentityLaw.md`
* `README.DeclaredVsMechanizedAudit.md`
* `README.MechanizationClosureGate.md`
* `README.PacketWorkflowChain.md`
* `README_DoorOneDevelopmentalOutline.md`
* `README.SemanticOverlayContract.v0.md`

Its purpose is narrower:

* define the next app-surface direction after the semantic overlay turnaround,
* stop treating the current composed browser surface as the long-lived destination,
* establish a thin router/home shell above dedicated viewers,
* preserve structural primacy while improving usability,
* and stabilize the next packet arc for live/static/inspection GUI development.

This note governs **app-surface transition posture** only.

It does **not** govern:

* runtime truth,
* canon activation,
* final GUI styling,
* final product/demo packaging,
* or a full public-facing application architecture by itself.

---

## 1. Why this note exists

The semantic overlay turnaround and cleanup waves cleaned the major runtime, report, workbench, and read-side seams enough that the next bottleneck is no longer semantic blur inside the active Door One stack.

The next bottleneck is **app-surface composition**.

The current browser surface is not wrong, but it is now carrying too many roles at once:

* execution shell
* inspection surface
* demo surface
* dense structural analysis surface
* overlay-bearing read-side surface

That composition was useful during the stabilization phase.

It is no longer the cleanest shape for the next stage.

The next lawful step is to stop treating the current composed surface as the main destination and instead treat it as a **temporary composition surface** while introducing:

* one thin **Home / Router shell**
* dedicated **Live**, **Static**, and **Inspection** viewers
* one shared structural viewer contract
* and clearly subordinate overlays

One-line summary:

**The next app stage should separate routing from viewing, and separate viewer purpose before adding more UI density.**

---

## 2. Core transition rule

The composed app surface should no longer be treated as the long-lived central face of Door One.

Instead:

* the top-level face becomes a **thin router/home shell**
* dedicated viewers become the real working surfaces
* the current composed environment becomes transitional legacy composition or inspection support
* structural evidence remains primary
* semantic/readiness/review overlays remain optional and subordinate

Corollary rules:

* bigger page is not automatically better architecture
* mixed panes are not automatically more usable
* operator convenience must not flatten layer separation
* viewer purpose should remain explicit
* display must not outrun mechanism

---

## 3. Transition goal

The app surface should move toward a **3-layer GUI structure**.

### Layer 1 — Home / Router shell

Purpose:

* choose source
* choose mode
* choose viewer family
* set overlay posture
* frame session context

The router shell should stay thin.

It should not become the dense inspection surface itself.

Its job is:

* source selection
* session framing
* route selection
* bounded posture visibility

It is a routing surface, not a truth surface.

### Layer 2 — Dedicated viewers

Each viewer should have one primary job.

Initial viewer split:

* **Live Structural Viewer**
* **Static Structural Viewer**
* **Inspection / Forensics Viewer**

These viewers may later contain multiple viewer families, but their purpose should remain distinct.

### Layer 3 — Toggleable overlays

The semantic overlay contract remains active:

* semantic overlay
* readiness overlay
* review overlay

These overlays must remain:

* late-bound
* removable
* subordinate to structure
* non-authoritative
* non-primary

No viewer should require overlays in order for structural evidence to remain legible.

---

## 4. Why the split is necessary

The current visuals already demonstrate more than one UI purpose.

### 4.1 Inspection purpose is already real

The bounded-state frames inspector is a dense inspection/forensics panel.

It is useful for:

* HI / MI structural state review
* replay/reconstruction posture
* retained support inspection
* bounded operator questioning
* explicit display-vs-mechanization discipline

That makes it a strong **Inspection Viewer** candidate.

### 4.2 Pipeline explanation purpose is already real

The structural pipeline graphic teaches a layered story:

* source fixture
* structural states
* retained signature
* replay / reconstruction objects
* interpretation overlay
* review gate

That is closer to:

* architectural explanation
* operator orientation
* or router-shell context

than to a dense inspection surface.

### 4.3 Live geometry/runtime viewing is a different job

A live structural viewer will need:

* animation
* streaming frames
* viewer-specific telemetry
* runtime timing visibility
* a different pacing and spatial layout than a dense audit panel

That should not be forced into the same surface as the bounded inspection panel.

One-line summary:

**Inspection, explanation, and live viewing are not the same job and should not be forced into the same main page.**

---

## 5. Target top-level app structure

The preferred top-level app structure is:

### 5.1 Home / Router shell

The only true top-level face.

Suggested functions:

* source selector
* mode selector:

  * Live
  * Static
* viewer-family selector
* overlay toggle
* posture summary:

  * source active
  * lineage available
  * overlay on/off
  * display below authority

This shell should remain light.

It should not render dense structural analysis content by default.

### 5.2 Viewer family routing

Initial viewer family routing should allow entry into:

* Spectral / Frequency-Time Viewer
* Energy / Amplitude Viewer
* Persistence / Recurrence Viewer
* Inspection / Forensics Viewer

This list may grow later, but the first transition should stabilize these roles before adding more.

### 5.3 Transitional legacy route

The current composed app may remain temporarily available as:

* legacy composed environment
* inspection bundle
* or transitional shell

It should not be treated as the final surface architecture.

## 6. Viewer split

## 6.1 Live Structural Viewer

Purpose:

* show real-time structural evolution
* visualize geometry, basins, trajectories, and perturbation in motion
* include runtime and render telemetry
* support live operator observation without semantic inflation

Live viewers must carry runtime telemetry because they are runtime-facing surfaces, not static inspection objects.

## 6.2 Static Structural Viewer

Purpose:

* inspect pre-rendered or bounded structural objects
* pause, compare, annotate, and study
* emphasize provenance and generation posture rather than runtime render metrics

Static viewers do not need live telemetry by default.

They are inspection-oriented bounded surfaces.

## 6.3 Inspection / Forensics Viewer

Purpose:

* dense bounded analysis
* retained support inspection
* replay / reconstruction contrast
* operator questioning
* structural audit posture

This is the home for the current dense state-inspector style.

It should remain available because it is already useful.

---

## 7. Initial viewer families

## 7.1 Spectral / Frequency-Time Viewer

This should be the first hero viewer.

It is the strongest candidate for:

* live geometric field viewing
* basin rendering
* trajectory lines
* perturbation/event markers
* structural evolution across time

This is the cleanest first live viewer because it aligns directly with the current structural signal stack.

## 7.2 Energy / Amplitude Viewer

This viewer should own:

* RMS / envelope
* energy redistribution
* peak / collapse / sustain behavior
* telemetry-aligned energy interpretation in live mode

This is also a natural place for some runtime/browser telemetry in early live work because it remains intuitive to read.

## 7.3 Persistence / Recurrence Viewer

This viewer should own:

* recurrence structure
* retained-vs-live continuity
* support persistence
* bounded memory/support patterns below semantic inflation

This is likely less flashy than the spectral viewer but deeply aligned with DME’s actual continuity goals.

## 7.4 Inspection / Forensics Viewer

The current HI / MI / replay / reconstruction / operator-questioning panels belong here.

They are not obsolete.

They are simply a different viewer purpose.

---

## 8. Shared structural viewer rule

All viewers must consume the **same structural truth**, projected differently.

This is a core rule.

Do **not** create:

* one truth contract per viewer
* one demo truth and one real truth
* one structural payload for “serious” use and one semantic payload for “nice” use

Instead:

* one structural stream or frame contract
* multiple lawful projections

This preserves:

* structural primacy
* inspection honesty
* bundle compatibility
* future maintainability

One-line summary:

**One structural stream, multiple lawful projections.**

---

## 9. Overlay rule inside viewers

Overlays remain:

* optional
* toggleable
* subordinate
* removable
* non-authoritative

A viewer with overlays turned off must still remain useful.

A viewer with overlays turned on must still preserve structural anchoring.

This means:

* overlays must not replace structural evidence
* overlays must not become the base layer
* overlays must not collapse readiness/review into runtime truth

---

## 10. Live telemetry rule

Live mode requires explicit telemetry.

At minimum, live surfaces should be designed to carry:

* pipeline latency
* render FPS
* dropped frame count
* browser jitter
* processing jitter
* queue depth or cadence if available

This is not decorative.

It is mechanization honesty.

Live mode is a runtime-facing surface and should make timing distortion visible rather than hiding it.

Static mode does not need the same telemetry posture.

---

## 11. Transition stages

## Stage A — App surface transition

* introduce Home / Router shell
* keep existing entry points temporarily
* demote current composed app to transitional legacy composition or inspection support

## Stage B — Shared viewer contract

* define one shared structural viewer payload
* define one live telemetry payload
* confirm overlay attachment rules for viewers

## Stage C — First viewer pair

* build Live Spectral Viewer v0
* build Static Spectral Viewer v0

## Stage D — Viewer expansion

* Energy Viewer
* Persistence Viewer
* Inspection Viewer migration from the current dense bounded-state panels

## Stage E — Overlay integration

* semantic overlay
* readiness overlay
* review overlay

All remain optional and subordinate.

---

## 12. What this plan is trying to prevent

This transition is designed to prevent:

* growing one giant mixed page forever
* routing logic and inspection logic collapsing into one surface
* semantic overlays becoming base viewer truth
* live runtime surfaces pretending to be static inspection surfaces
* static audit panels pretending to be real-time field viewers
* UI growth outrunning mechanized structural boundaries

## 13. What is now true

* The current composed browser surface was useful for the stabilization phase.
* The next clean step is separation by purpose, not expansion by accumulation.
* A thin router shell above dedicated viewers is the cleanest next lawful app move.
* The first hero viewer should likely be Spectral / Frequency-Time.
* The current dense panels are best preserved as an Inspection / Forensics Viewer rather than treated as the main long-lived home surface.
