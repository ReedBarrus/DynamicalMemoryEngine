- structural transition vocabulary,

- test distinctions at the read-side first,
- add the smallest lawful derived surface when flattening appears,
- keep ambiguity visible,
- avoid prematurely rewriting operators or ontology.

However, as the probe family grows, the project needs one compact accounting surface that answers:

- what each probe is for,
- what distinction it preserves,
- what seam it belongs to,
- what it depends on,
- what it must not decide,
- and whether it is now:
  - active,
  - comparison-only,
  - closed enough,
  - deferred,
  - or a future operator-admission candidate.

This note exists to provide that accounting surface.

## 3. Core accounting rule

A probe is not “sprawl” merely because it is numerous.

A probe is lawful when it does one or more of the following:

- preserves a real distinction that the current surfaces flatten,
- exposes ambiguity honestly,
- clarifies lens dependence,
- answers a real diagnostic question,
- prevents ontology leakage,
- or prevents premature operator expansion.

A probe should be considered for consolidation only when:

1. its distinction is already stably preserved elsewhere, or
2. it has become a repeated support surface with the same role across multiple families, or
3. it no longer adds new audit value for the current Door.

Until then, bounded probe plurality is preferable to hidden semantic inflation.

## 7. Probe admission / consolidation rule

A probe should remain probe-only when:

- it is still primarily clarifying whether a distinction is real,
- it is strongly lens-dependent,
- it remains exploratory or family-specific,
- or its output would create hidden authority if admitted upstream.

A probe may become a standard read-side report surface when:

- it has repeated utility across multiple families,
- its support basis is stable and legible,
- it reduces ambiguity without inflating ontology,
- and it remains clearly below runtime authority.

A probe may become a future operator-admission candidate only when:

1. the distinction survives across multiple relevant families,
2. the correction cannot be handled honestly as a downstream support surface,
3. runtime/operator code is actually responsible for the missing distinction,
4. distortion remains materially high if kept downstream,
5. and admission would still be the lowest lawful layer.

Until those conditions are met, probe-only is the preferred posture.
