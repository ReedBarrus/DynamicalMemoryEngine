## 1. Why this note exists

The first semantic overlay turnaround and Cleanup Wave A succeeded in removing the main semantic blur from the active Door One runtime and read-side stack.

The remaining issues are no longer major structural contamination problems.

They are mostly:

* transitional compatibility residue,
* hot naming surfaces,
* boundary-ambiguous helper fields,
* and a few still-warm downstream review/routing labels.

Those issues are easy to lose track of precisely because they are smaller than the original architectural problem.

This note exists so they do not disappear into conversation residue or get forgotten between packet waves.

One-line summary:

**The main semantic overlay turnaround is complete enough; what remains is controlled residue management, not structural rescue.** 

---

## 2. Current posture

The drift audit established that:

* the main turnaround did **not** leave a hidden second semantic architecture,
* the highest remaining risk is no longer in structurally primary runtime,
* and most surviving hotspots are compatibility residue or naming debt rather than active structural dishonesty. 

Cleanup Wave A then burned down the three hottest residual seams:

1. `interpretation.trajectory` stopped being an active required bridge for cleaned readiness/dossier consumers
2. `coordination_hints.pre_commitment` stopped being an active downstream signal
3. review-routing posture around `promotion_recommendation` / `consensus_review` was hardened so it no longer feels promotive by default

That leaves a smaller, more stable cleanup backlog.

## 3. Backlog classes

This backlog uses four practical route classes.

### 3.1 Keep transitional

The item is currently useful as bounded transitional residue and is reducing drift rather than increasing it.

### 3.2 Rename later

The item is lawful enough now, but its name is hotter, broader, or clumsier than the seam justifies.

### 3.3 Fence later

The item is still lawful, but its wording or role remains warm enough that another bounded tightening pass is likely justified.

### 3.4 Retire on migration completion

The item is now mostly residue. It should be removed once remaining consumers no longer need it.

## 8. Working rule for future cleanup

Use this backlog only for **bounded cleanup packets**.

Do not open a cleanup packet merely because a name feels ugly.

Open one only when the item presents real risk of:

* compatibility becoming architecture,
* support drifting into memory language,
* review drifting into approval,
* or wording outrunning current support/mechanization.

One-line rule:

**Clean up heat where it can still mislead; tolerate residue where it is explicit, bounded, and temporary.**
