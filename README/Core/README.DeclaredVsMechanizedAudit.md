# Dynamical Memory Engine — Declared vs Mechanized Audit

## Status

This note is a structural audit reference.

It exists to prevent a specific distortion class:

**declared object or surface shape being mistaken for actual mechanized capability.**

This note does **not** override:

- `README_MasterConstitution.md`
- `README_WorkflowContract.md`
- `README_RepoPlacementConstitution.md`

- replay / request / retention / canon / consultation notes already in the repo

This note is a **clarifying audit instrument**.

Its purpose is to identify where DME currently has:

- declared posture only,
- displayed surface only,
- partially mechanized function,
- or fully mechanized function.

It also defines the acceptance horizons required before stronger claims about stabilization, consultation, replay, retention, query, or canonization are justified.

---

## 1. Why this note exists

DME now has enough surfaces, helper models, request objects, replay objects, and app composition that some capabilities can appear more complete than they actually are.

This is not necessarily deception.
It is often the result of healthy architectural progress:

- constitutional posture becomes clear,
- UI surfaces become legible,
- object models become coherent,
- and only later does the full backend mechanism arrive.

That is normal.

The risk is that **surface clarity can outrun mechanism**.

This note exists to prevent that drift.

---

## 2. Core rule

**Declared is not mechanized.  
Displayed is not mechanized.  
Prepared is not fulfilled.  
Candidate is not canon.  
Replay-shaped is not reconstructed.**

Any capability in DME should be understood through three distinct questions:

1. Is it declared?
2. Is it displayed?
3. Is it mechanized?

Only the third should be treated as actual functional attainment.

---

## 3. Audit categories

### 3.1 Declared

The capability has explicit constitutional language, object shape, posture, or intended interface, but no actual implementation path yet.

### 3.2 Displayed

The capability has visible UI or read-side representation, but may still be powered only by local state, static shaping, or incomplete backend logic.

### 3.3 Mechanized

The capability has an actual bounded implementation path that performs the intended function under the declared constraints.

### 3.4 Partially mechanized

The capability has some real implementation, but still depends on manual handoff, incomplete backend support, or missing enforcement.

## 5. Distortion classes to watch for

The following distortions are now recognized project risks.

### 5.1 Replay declaration mistaken for replay execution
Replay object or replay panel exists, so replay is assumed mechanized.

### 5.2 Candidate/review posture mistaken for canon path
Readiness + dossier + request prep is assumed to equal canonization.

### 5.3 Query artifact mistaken for query engine
Presence of `q` is assumed to equal broad structural querying.

### 5.4 Retained-tier language mistaken for retention mechanics
Tier labels are assumed to correspond to actual durable storage and retrieval.

### 5.5 Consultation request mistaken for consultation fulfillment
Prepared request is assumed to mean the review loop exists.

### 5.6 Tandem summary mistaken for full evidence access
Cross-surface parity shaping is assumed to equal deep inspection.

### 5.7 Adapter normalization mistaken for source-family understanding
Import success is assumed to equal semantic correctness.

## 6. Acceptance horizons

This section exists so “working harness” is not mistaken for “stabilized system.”

## Horizon A — Harness alive

Required:
- runtime executes
- shell executes
- workbench assembles
- lab HUD displays
- request object prepares
- replay object prepares

Interpretation:
The field is populated.
This is not stabilization.
This is basic operational aliveness.

---

## Horizon B — Structural coherence

Required:
- ingest lineage is preserved into read-side surfaces
- shell, HUD, tandem, request, replay agree on core object identity
- adapter-mediated inputs enter the same lawful runtime path
- stale-shape drift is largely removed
- failure paths remain explicit and non-filling

Interpretation:
The current system is structurally coherent enough to inspect and debug meaningfully.

---

## Horizon C — Replay honesty

Required:
- provenance reconstruction pipeline exists
- replay trace is emitted
- replay trace respects declared lens
- replay trace respects retained tier
- replay does not rerun operators
- replay failure is explicit
- replay fidelity and replay honesty can be measured

Interpretation:
Replay becomes a real system capability instead of a declared surface shape.

---

## Horizon D — Retention honesty

Required:
- Tier 0 and Tier 1 mechanics exist
- replay can distinguish live vs durable state materially
- retrieval respects retained tier
- pruning/retention behavior is inspectable
- retention posture is no longer merely declared

Interpretation:
The system begins to support durable provenance and bounded replay claims.

---

## Horizon E — Consultation / query realism

Required:
- structural query engine exists beyond current-run `q`
- cross-run/session queries are bounded and inspectable
- consultation preparation has a real fulfillment protocol
- request resolution objects or equivalent review outputs exist

Interpretation:
The system begins to support real navigability and real consultability.

---

## Horizon F — Canon handoff readiness

Required:
- replay honesty exists
- retention honesty exists
- consultation protocol exists
- constitutional guardrails are mechanized
- explicit review acceptance protocol exists
- authority seam between runtime/read-side and canon is mechanically enforced

Interpretation:
Canonization may be considered structurally, but is still not implied.

## 7. Immediate active priorities

For the current leg, the highest-value priorities are:

1. provenance reconstruction pipeline

## 8. What this note forbids

This note forbids treating the following as equivalent:

- replay object = replay backend
- request object = consultation protocol
- dossier = canon
- query artifact = query engine
- tier label = retention system
- composed app = backend maturity

DME currently has a live operational harness and several mechanized runtime and read-side seams, but many higher-order surfaces remain declared or only partially mechanized; this note exists to prevent surface clarity, posture language, or prepared objects from being mistaken for fully implemented structural capability.
