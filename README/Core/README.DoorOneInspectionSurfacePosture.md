## 1. Constitutional posture

Door One remains below canon.

Inspection surfaces in Door One are:

- read-side only,
- non-authoritative,
- evidence-preserving,
- bounded by runtime law,
- forbidden from turning display into truth, canon, ontology, or promotion.

This note exists because a surface may remain technically lawful while still encouraging users to over-believe higher-order summaries more strongly than the underlying evidence justifies.

The risk addressed here is not runtime illegitimacy.
The risk is read-side semantic drift.

---

## 2. Problem statement

As Door One accumulates more lawful outputs, inspection surfaces begin to compress unlike evidence planes into one glance.

Examples include:

- provenance identifiers,
- runtime counts and audit facts,
- structural and interpretive summaries,
- readiness labels,
- dossier packaging,
- bounded consensus posture.

This can create a false feeling of authority even when the underlying code remains lawful.

The danger is not that Door One has become canon.
The danger is that the human eye begins to treat interpretive or review-adjacent framing as if it were more authoritative than provenance and runtime evidence.

---

## 3. Primary inspection rule

Every Door One inspection surface must preserve this visual priority order:

1. Provenance
2. Runtime Evidence
3. Interpretation
4. Review Surfaces

This is a read-side discipline only.
It does **not** create a new layer.
It does **not** alter artifact meaning.
It does **not** promote any output upward.

---

## 4. Evidence plane definitions

### Plane 1 — Provenance

This plane should remain topmost or otherwise impossible to miss.

Its role is to preserve where the viewed material came from.

Typical contents:

- source mode
- source identity
- stream identity
- channel / modality
- policy references
- cycle / run label
- time span
- receipt identifiers or references

This plane should answer:

- what source produced this?
- what stream am I looking at?
- what policy regime shaped entry?
- where would I look for receipts?

### Plane 2 — Runtime Evidence

This plane is the primary body of Door One inspection.

Its role is to show non-interpretive execution facts and structural runtime organization.

Typical contents:

- artifacts
- substrate counts
- segment boundaries
- transition counts / shares
- skipped windows
- merge failures
- anomaly counts
- runtime audit summaries

This plane should answer:

- what actually happened in runtime?
- what structure was produced?
- what warnings or gaps exist?
- what evidence supports the visible state?

### Plane 3 — Interpretation

This plane is allowed, but must be visibly downstream.

Its role is to provide bounded derived readouts over runtime evidence.

Typical contents:

- trajectory character
- attention / memory readouts
- motion labels
- continuity labels
- recurrence labels
- other explicitly bounded derived summaries

This plane should answer:

- how is the system characterizing the evidence?
- what bounded interpretation is being offered?

This plane must be explicitly labeled as derived, bounded, and non-authoritative.

### Plane 4 — Review Surfaces

This plane is the lowest-priority plane in Door One.

Its role is to expose bounded review posture without implying canon, truth, or promotion.

Typical contents:

- promotion readiness
- canon candidate dossier
- bounded consensus review posture
- blockers / insufficiencies
- candidate claim packaging

This plane should answer:

- how has the current evidence been packaged for review?
- what bounded review posture exists?
- what remains blocked or insufficient?

This plane must not imply:

- canon activation
- trusted truth
- promotion outcome
- ontology
- system-level authority

## 5. Visual posture rules

Inspection surfaces should follow these presentation rules when feasible.

### Rule 1 — Provenance first

Provenance should be persistently visible and visually prior to interpretation or review posture.

### Rule 2 — Runtime evidence outranks interpretation

Non-interpretive runtime facts should carry greater visual weight than derived labels.

### Rule 3 — Review surfaces are fenced

Promotion readiness, dossier, and consensus posture should be visibly separated from runtime evidence and interpretation.

Review surfaces should be lower prominence and may be collapsed by default.

### Rule 4 — Labels should link back to evidence

Where possible, interpretive or review labels should be expandable or traceable back to the runtime evidence that produced them.

### Rule 5 — Audit should remain visible

Audit visibility is part of runtime honesty.
Inspection surfaces should not hide skipped windows, merge failures, or other bounded runtime warnings behind stronger summary labels.

### Rule 6 — Display is not authority

No visual treatment may imply that a surface has become truth, canon, ontology, or promotion simply because it is compact, attractive, or easy to read.

---

## 6. Lawful read-side behavior

Inspection surfaces may:

- shorten IDs for readability,
- restyle evidence for clarity,
- choose read-side projection layouts,
- group related evidence planes,
- collapse low-priority sections,
- expose drill-down paths into supporting evidence.

Inspection surfaces may not:

- alter runtime data,
- alter workbench meaning,
- create hidden authority,
- treat styling as evidence,
- treat query state as truth,
- infer canon from visibility,
- infer ontology from substrate organization,
- promote based on review convenience.

---

## 7. Drift questions for review

When auditing any Door One inspection surface, ask:

1. What does this surface preserve?
2. What does this surface tempt me to over-believe?
3. Is provenance easier to miss than review posture?
4. Are runtime facts visually weaker than interpretive labels?
5. Are review surfaces visually stronger than their actual authority?
6. Can a user trace higher-order labels back to evidence?
7. Does this surface remain clearly below canon?

If these questions cannot be answered cleanly, tighten the inspection surface before adding new semantics.

---

## 8. Practical implications for current Door One HUDs

For the browser structural memory HUD, the preferred structure is:

- Provenance
- Runtime Evidence
- Interpretation
- Review Surfaces

For terminal or script-facing summaries, compactness is allowed, but provenance and runtime honesty should remain visible before review-adjacent framing whenever feasible.

For future replay/digest readers, provenance lineage and runtime references should remain primary over convenient interpretive packaging.

## 10. Current recommendation

Before adding more interpretive density or review packaging to Door One inspection surfaces:

1. strengthen provenance visibility,
2. strengthen runtime and audit visibility,
3. visibly demote review posture,
4. preserve traceability from labels back to evidence.

That is the preferred tightening move for the current phase.
