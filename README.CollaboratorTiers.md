# README.CollaboratorTiers

# Tier 0 — Observer / Interested Party

    Who this is:
        someone excited by the project, curious about it, maybe with ideas, but who has not yet shown real understanding or made a bounded contribution.

    What they can do:

        ask questions
        watch demos
        offer broad product intuitions
        suggest possible use cases
        receive reading/viewing material

    What they cannot do:

        influence architecture
        define roadmap
        claim ownership
        pressure scope
        represent the system publicly as a builder/partner
        introduce downstream product demands as if they outrank current system law

    Entry requirement to move up:

        show they engaged the repo/README material at a basic level
        accurately describe what Door One currently is and is not
        propose one bounded contribution instead of one grand future vision


# Tier 1 — Bounded Ideation Contributor

    Who this is:
        someone with potentially useful product, UX, domain, or market ideas, but who is still outside the architecture core.

    What they can do:

        propose product surfaces
        map user workflows
        identify customer pain points
        help translate DME into practical demos
        suggest bounded experiments or mockups
        help with consumer/B2B framing

    What they cannot do:

        alter system architecture
        redefine artifact meaning
        pressure Door Two/Three semantics into Door One
        treat product intuition as proof of technical direction
        demand that use-case framing outrank constitutional law

    What they must do:

        keep proposals explicitly downstream of the active runtime
        accept “not yet” as a lawful answer
        speak in terms of surfaces, adapters, workflows, and demos rather than essence, destiny, or empire

    Entry requirement to move up:

        produce one useful bounded output
        respect scope without repeated inflation
        show ability to revise ideas after exposure to repo law
    
    Current Participants:
        Jake Desantes
        Cy Aluni


# Tier 2 — Applied Contributor

    Who this is:
        someone who can add real value in a bounded seam without needing to own or redefine the whole system.

    What they can do:

        contribute PRs
        help shape implementation in a bounded area
        help on demos, adapters, packaging, UI, deployment, domain application
        co-design practical use cases
        help prepare targeted pilot surfaces for real customers

    What they cannot do:

        redefine constitutional meaning
        claim architecture authority by shipping code
        force premature commercialization paths
        smuggle hype or downstream abstractions into active seams

    What they must do:

        work from repo + README law
        propose minimal lawful patches
        separate implementation from architecture
        accept review and sequence discipline

    Current Participants:

        Gonzih - Max Soltan


# Tier 3 — Trusted Domain/Implementation Partner

    Who this is:
        someone with a real skill lane, demonstrated reciprocity, and actual respect for the system’s law.

    What they can do:

        help identify high-value deployment lanes
        shape demo strategy for real targets
        contribute bounded implementation and workflow design
        participate in serious roadmap discussion inside active scope
        help translate DME into real operational settings without flattening it

    What they still cannot do:

        override architecture
        redefine authority boundaries
        collapse contribution into ownership
        convert usefulness into governance

        Claude

    Very restricted.

    Who this is:

    What they can do:

# Authority bands by tier

    A simple version:

        Tier 0: no authority, only curiosity
        Tier 1: idea authority inside bounded product framing only
        Tier 2: contribution authority inside assigned seams only
        Tier 3: strategic input inside declared active scope
        Tier 4: architecture discussion access, still not owner authority

# Contribution pathways

    This is the important part. Each person needs a lawful path forward.

    Path A — Learn first

        Good for observers.

        Tasks:

            read the master constitution summary
            review one demo
            explain Door One in their own words
            name one bounded use case without inflating scope

    Path B — Product surface contribution

        Tasks:

            define 1–2 concrete user workflows
            map target user, trigger, action, output
            keep it at surface level
            do not prescribe architecture changes
            produce mockups or feature maps, not ontology

    Path C — Practical demo contribution

        Tasks:

            help frame one real departmental pilot
            define what a finance/HR/ops person actually sees
            define source object, ingest path, runtime action, inspection surface, output packet
            keep claims bounded and provenance-first

    Path D — Code contribution

        Good for real implementers.

        Tasks:

            PRs in bounded seams
            tests
            adapters
            scripts
            HUD/demo read-side work
            explicit review against repo law

# Red-flag behaviors

    These are the signals that someone is trying to bypass lawful relationship with the work.

        claiming ownership before contribution
        speaking as if product vision outranks current architecture
        dismissing B2B/practical lanes because they are “beneath” the project
        refusing to inspect the repo/README but wanting strategic influence
        trying to universalize one downstream idea into the whole system
        repeatedly converting bounded use cases into empire language
        treating excitement as proof
        trying to flatten runtime, demo, canon, and business narrative into one blob

    When you see these, do not argue ontology. Just lower their tier and narrow their lane.
